## Open Risks

- Avoid reverting or normalizing unrelated user changes, even though the current worktree no longer looks broadly dirty.
- Without regular updates, Codex state can drift back into chat-only context and stop being durable.
- `zig build test` is not hermetic yet: the real-asset scene regressions require the canonical extracted asset tree, and `port/build.zig` still hard-fails if the repo-local SDL2 layout is missing.
- Linux Bash is currently a research/editing environment, not the canonical validation host. During the 2026-03-26 assessment, `zig build test`, `zig build tool -- inspect-scene 2 --json`, and `zig build run` all timed out from Bash while the supported dependency path and helper scripts still targeted Windows.
- The player actor target for `SCENE.HQR[2]` is only partially locked: slot `0` is proven, but a direct hero body/animation binding is still unresolved and should not be guessed in phase 1 code.
- Giver semantics are still intentionally conservative: the loader can expose allowed bonus kinds from `Info0`, but the final spawned bonus remains runtime-dependent through `WhichBonus()` and should not be collapsed into one canonical load-time value without stronger evidence.
- A few track opcodes still lack explicit layout coverage in the checked-in runtime switch (`TM_NOP`, `TM_DO`, and `TM_AFF_TIMER` remain unproven there, while `TM_REM` appears in real assets without a matching `DoTrack` case), so future work should not inflate their semantics beyond the current structural byte-level treatment without stronger evidence.
- Life-script interpretation is still not wired into the scene surface. The offline decoder plus broader audit now prove that current real scene data only reaches unsupported `LM_DEFAULT` and `LM_END_SWITCH`, the switch-family source pass still does not prove structural handling for either id, and the other six named unsupported `LM_*` ids remain unproven even though they do not appear in the current asset tree, so scene integration remains blocked on stronger source-backed evidence rather than parser mechanics alone.
- The canonical `SCENE.HQR[2]` / `LBA_BKG.HQR[2]` room pairing can still tempt direct-entry assumptions, but the interior loader actually depends on `LBA_BKG.HQR[0]` plus the late `TabAllCube` indirection entry, so future background work should not overfit to entry `2` bytes alone.
- The HQR helper surface is now intentionally split: existing `extractEntryToBytes` / `decodeEntryToBytes` still follow the repo's older one-based view that skips classic slot `0`, while interior background work must use `extractClassicEntryToBytes` / `decodeClassicEntryToBytes` for loader-faithful `LBA_BKG.HQR` indices. Mixing those spaces will silently point at the wrong payload.
