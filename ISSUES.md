# Common Agent Traps

- A stale ignored `port/hello-world/` directory can survive on disk after the real Phase 1 workspace is moved to `port/`. If Phase 1 docs claim the placeholder was replaced but the directory still exists locally, delete it instead of preserving the obsolete layout.
- `sidequest/` and `LM_TASKS/` are independent workstreams, not part of canonical memory pickup. Do not treat them as execution owners unless a prompt explicitly widens scope.
- `python3 tools/codex_memory.py context --include-history ...` still hides excluded durable history by default. If a task explicitly needs `sidequest/`, `LM_TASKS/`, LM/debug streams, or other noncanonical history, add `--include-excluded-history` instead of assuming the records disappeared.
- The checked-in memory files can lag behind the live worktree. Verify the code and `git status` before treating `docs/codex_memory/current_focus.md` or any subsystem pack as a statement about dirtiness, generated outputs, or active experiments.
- `docs/PORTING_REPORT.md`, `docs/MEMSYS.md`, and older migration notes are historical context, not the current implementation contract. Use `docs/LBA2_ZIG_PORT_PLAN.md`, `docs/codex_memory/`, and the live `port/` tree for current state.
- `docs/PROMPT.md` is volatile. Cross-check it against `current_focus.md`, the relevant subsystem packs, and recent typed history before treating it as the next canonical slice.
- Raw `SCENE.HQR` entry bytes are not the runtime scene payload. The classic loader reads an HQR `COMPRESSED_HEADER` first and may LZ-decompress the entry before parsing scene fields, so typed decoders must decode the resource block before interpreting world/object/zone data.
- Scene numbering is easy to misread. The classic loader calls `Load_HQR(..., numscene + 1)` because `SCENE.HQR[0]` is reserved for `SizeCube.MAX`, while the current `inspect-scene` command accepts the raw one-based HQR entry index. Keep semantic scene numbers and HQR entry numbers separate.
- The raw-to-loader mapping has two offsets in this repo, not one. Classic code uses raw HQR slot `0` for `SizeCube.MAX`, while the Zig tooling exposes physical entries starting at `1`, so `SCENE.HQR[2]` is classic loader scene `0` and `classic_loader_scene_number` must use `entry_index - 2` for real scenes.
- The MBN `SCENE.HQR` lists use type markers that matter. In post `177798`, `~1` means isometric room data and `~2` means outside scene data; keep the type marker with the descriptor instead of copying only the human-readable scene name.
- The checked-in MBN corpus index is broader than the raw evidence layer. A topic can appear in `docs/mbn_reference/corpus/index/topics_discovered.jsonl` and `topic_classification.jsonl` without a matching raw post payload or any `tools/mbn_workbench.py search-evidence` hits.
- The shell-detection snippet in `AGENTS.md` is misleading under Bash because it uses PowerShell redirection syntax on the Bash side. Treat the current shell as Bash from the provided environment context or use a Bash-safe probe instead.
- On the Windows-first build path, `bash -lc` is not a reliable wrapper for canonical Zig verification. Use `py -3 .\scripts\dev-shell.py exec --cwd port -- zig build ...` or open `py -3 .\scripts\dev-shell.py shell` from native PowerShell for real validation.
- On the current Windows setup, `bash -lc python3 ...` works for repo scripts but that interpreter does not have `pip`. Use native PowerShell `py -3 -m pip install -r requirements.txt` for Python dependency installs, then keep using `python3` or `py -3` for the actual script/test commands as needed.
- In native PowerShell, piping `zig build ...` through `Out-String` can still misreport a successful build as exit code `-1`. Keep scripted acceptance on `scripts/verify_viewer.py` plus `scripts/dev-shell.py exec` instead of rebuilding native-command capture in ad hoc PowerShell.
- Windows `vcvars` imports can report the toolchain path under `Path` instead of `PATH`. Keep `scripts/dev-shell.py` normalizing environment keys case-insensitively before looking up tools like `cl`, `cmake`, and `zig`.
- The guarded runtime/load seam currently has one positive pair: `19/19`. Keep `2/2`, `44/2`, and `11/10` as guarded negative cases, and use the explicit unchecked test-only loader if you need `11/10` fragment evidence.
- The guarded `19/19` viewer zone summary intentionally splits presentation. HUD copy uses `ZONES NONE` / `ZONES <indices>`, while stderr diagnostics use structured `zones=none` / `zones=<indices>`. Keep both driven from runtime-owned zone membership instead of collapsing them into one invented current-zone label.
- The canonical `SCENE.HQR[2]` / `LBA_BKG.HQR[2]` pairing is not the whole interior background loader story. Classic interior loading also depends on `LBA_BKG.HQR[0]` as `T_BKG_HEADER` plus a late `TabAllCube` entry at `BkgHeader.Brk_Start + BkgHeader.Max_Brk`.
- `port/src/assets/hqr.zig` now has two HQR index spaces on purpose. The older helpers follow the repo's one-based view that skips classic slot `0`, while the classic helpers use the engine's zero-based `Load_HQR` numbering. Use the classic helpers for `LBA_BKG.HQR[0]` and other loader-faithful paths.
- `zig build test` is a real-asset regression gate, not a hermetic unit-test-only pass, and it does not prove the executable targets compile. Keep explicit `zig build run` or `zig build tool` acceptance commands when runtime code changes.
- Tool-only CLI paths can stay half-wired if you only trust `zig build test-fast` or `zig build test`. A new subcommand can still be missing enum/parse/run plumbing, or formatter helpers can still be unreachable, until you execute the real `zig build tool -- <command> ...` path.
- Interrupted viewer launches can leave `port/zig-out/bin/lba2.exe` locked by a stale `lba2.exe` process. If the next install step fails with `AccessDenied`, clear the stale process before treating it as a code regression.
- Rich guarded `19/19` diagnostics do not mean Phase 5 has a playable-path candidate yet. The supported startup still lands on `raw_invalid_start`, and the current scene metadata reports `track_count=0`, so future Phase 5 planning must not mistake the diagnostic baseline for a gameplay-ready slice.
- The new decoded-interior ranking surface is offline-only. `rank-decoded-interior-candidates` currently puts `SCENE.HQR[219]` first and `SCENE.HQR[19]` at `49/50`, but `inspect-room 219 219` still fails with `InvalidFragmentZoneBounds`, so do not equate the ranking winner with a runtime-admitted scene.
- The `219/219` blocker explanation now lives on `inspect-room`'s failure path. Before adding another blocker-only CLI or prompting for a new explanation surface, check whether stderr already prints `reason=invalid_fragment_zone_bounds` plus `event=fragment_zone_validation_issue ...` lines for the failing pair.
- `triage-same-index-decoded-interior-candidates` can return a technically compatible pair that says nothing about fragment-bearing room admission. On current assets, `86/86` is the highest-ranked compatible same-index pair above baseline, but it has `fragment_count=0` and `grm_zone_count=0`; `187/187` is the first compatible pair with actual fragment-zone data.
- `tools/codex_memory.py` validates the whole v2 tree, including subsystem-pack byte budgets, before it appends history. If an `add-*` command fails unexpectedly, check pack budgets and JSONL canonicalization, not just the new record.
- Memory history lookup now treats repo-relative `evidence_refs` as real retrieval edges too: exact and prefix path queries can match them, and subsystem-scoped pickup can infer ownership from them. If a record is still missing, check the query path string and normalization before assuming evidence-only history is ignored.
- `scripts/dev-shell.py shell` launches a configured child PowerShell; it does not mutate the parent shell in place. For scripted validation, prefer `scripts/dev-shell.py exec --cwd port -- ...` so the intended environment and working directory are explicit.
- Historical LM/debug notes and older typed history can still mention `scripts/trace-life.ps1`, but the canonical original-runtime tracer entrypoint is now `py -3 .\tools\life_trace\trace_life.py`. Do not reintroduce a wrapper layer unless the Python CLI is missing a required current-state behavior.
- `tools/life_trace/trace_life.py --mode tavern-trace` no longer accepts `--frida-repo-root`; it now uses `frida-agent-cli` through `--fra-repo-root` and that repo's `.venv` launcher. `basic` and `scene11-pair` still use direct Frida until the scene-11 proof lane is ported.
- Passing a save path as `ArgV[1]` to the original Windows `LBA2.EXE` is not enough evidence that the live scene has really loaded. On the current scene-11 proof lane, the staged `scene11-pair.LBA` spawn still times out before the fingerprint matches, so treat manual load-and-settle as required until a checked-in live run proves otherwise.
- `current.lba` is mutable runtime state on the original Windows lane. If you copy a staged save like `scene11-pair.LBA` into `current.lba`, later resume tests can rewrite it immediately, so re-stage it before each controlled repro and do not assume `Resume Game` still points at the staged bytes. On the current scene-11 proof lane, Frida attached across the later resume/load transition can also trigger an `APPCRASH`, so separate no-Frida control navigation from late attach until that boundary is understood.
- `SCENE.HQR` giver `Info0` is a bonus-flag mask that `WhichBonus()` resolves later from runtime state, so a typed decoder should expose allowed kinds or preserve the raw mask instead of inventing one singular `bonus_kind`.
- `SCENE.HQR[19]` zone bounds are trigger volumes, not admitted floor-truth mapping anchors. Do not invent runtime mapping policy from camera or zone geometry without new checked-in evidence.
- On the guarded `19/19` locomotion seam, the exact containing-zone membership for the admitted `39/6` fixture and the accepted south step is the empty set. Treat that as the checked-in truth for now; do not invent a singular current zone or a different coordinate scale to force a non-empty answer.
- The guarded `19/19` admitted footing intentionally splits presentation. HUD copy uses `SURF ...`, while stderr diagnostics use `current_footing=...`; both come directly from runtime-owned `local_topology.origin_surface` plus `origin_standability`, and they stay absent on raw-start / `origin_invalid`.
- Guarded unsupported-life load diagnostics report the first blocking blob, not a full scene inventory. For `11/10`, the canonical guarded hit is object `12` `LM_DEFAULT @ 38`, even though a later object still reaches `LM_END_SWITCH`.
- The decoded scene object array is intentionally shorter than `object_count`. The classic header count includes the reserved hero slot, so use `scene.objects.len` for actual object iteration and render sizing.
- `COMMON.H` is not sufficient proof for track or life opcode semantics. Let checked-in decoder behavior, asset-backed probes, and recovered source behavior beat header names or comments.
- Preserved legacy specs such as LBArchitect are useful structure guides, not byte-perfect numeric or indexing oracles. Let current asset-backed probes and tests win for exact values.
