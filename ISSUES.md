# Common Agent Traps

- A stale ignored `port/hello-world/` directory can survive on disk after the real Phase 1 workspace is moved to `port/`. If Phase 1 docs claim the placeholder was replaced but the directory still exists locally, delete it instead of preserving the obsolete layout.
- Raw `SCENE.HQR` entry bytes are not the same thing as the runtime scene payload. The classic loader reads an HQR `COMPRESSED_HEADER` first and may LZ-decompress the entry before parsing scene fields, so typed decoders must decode the resource block before interpreting world/object/zone data.
- Scene numbering is easy to misread. The classic loader calls `Load_HQR(..., numscene + 1)` because `SCENE.HQR[0]` is reserved for `SizeCube.MAX`, while the current `inspect-scene` command accepts the raw one-based HQR entry index. Keep semantic scene numbers and HQR entry numbers separate, and do not revive the old `SCENE.HQR[4]` exterior misclassification just because that entry decodes cleanly.
- The raw-to-loader mapping has two offsets in this repo, not one. Classic code uses raw HQR slot `0` for `SizeCube.MAX`, while the Zig tooling exposes physical entries starting at `1`, so `SCENE.HQR[2]` is classic loader scene `0` and `classic_loader_scene_number` must use `entry_index - 2` for real scenes.
- The MBN `SCENE.HQR` lists use type markers that matter. In post `177798`, `~1` means isometric room data and `~2` means outside scene data; if you pick a golden exterior target from that list, keep the type marker with the descriptor instead of copying only the human-readable scene name.
- The checked-in memory files can lag behind the actual worktree state. Verify the real worktree before treating `docs/codex_memory/current_focus.md` or any subsystem pack as a claim about dirtiness, generated outputs, or active local experiments.
- The checked-in MBN corpus index is broader than the raw evidence layer. A topic can appear in `docs/mbn_reference/corpus/index/topics_discovered.jsonl` and `topic_classification.jsonl` without a matching `docs/mbn_reference/corpus/raw/posts/<topic>.jsonl` payload or any `tools/mbn_workbench.py search-evidence` hits. If a forum thread lookup only finds index metadata, treat it as cataloged-but-unscraped and do not claim the repo preserves the post bodies.
- `docs/PROMPT.md` or older architecture notes can reference docs that are not actually present in the worktree. Verify the file exists before treating it as canonical context, and fix the reference instead of assuming the architecture doc set is complete.
- The shell-detection snippet in `AGENTS.md` is misleading under Bash. `pwsh -c 'Write-Host "PowerShell"' 2>$null || echo "Linux Bash"` uses PowerShell redirection syntax on the Bash side, so running it literally from Bash emits `/bin/bash: $null: ambiguous redirect` before falling through. Treat the current shell as Bash from the provided environment context or use a Bash-safe probe instead of trusting that exact command output.
- On the Windows-first build path, `bash -lc` is not a reliable wrapper for canonical Zig verification. In this repo the Linux helper shell can miss the Windows Zig toolchain even when native PowerShell is healthy, so run `.\scripts\dev-shell.ps1` and then `zig build test`, `zig build run`, or `zig build tool ...` from PowerShell for real workspace validation. Reserve Bash for repo inspection and source-oriented commands unless you have confirmed the required Windows toolchain is on the Bash-side `PATH`.
- `docs/PORTING_REPORT.md` still carries a historical `Current Workspace Status` section that talks about an earlier CMake-first setup path and environment steps. Treat that report as feasibility and evidence context only; use `docs/LBA2_ZIG_PORT_PLAN.md`, `docs/codex_memory/`, and the live `port/` tree for the actual current implementation state.
- `docs/PROMPT.md` can lag behind the durable memory and still point at a slice that the v2 subsystem packs or `docs/codex_memory/task_events.jsonl` already make obsolete. Before executing the prompt literally, cross-check it against `current_focus.md`, the relevant subsystem packs, and the latest typed history so you do not repeat completed work as if it were still the next step.
- The viewer's current brick patterns are synthetic probes derived from brick ids, not decoded `BRK` art. If the repo already has `drawBrickProbe`-style overlays, do not claim that actual brick-backed rasterization or preview evidence is done; check whether any code beyond header fields reads `BRK` payloads first.
- Older prompt text can overstate what `inspect-room --json` exposes for fragment-bearing rooms. The current CLI is now boundary-guarded and still reports only the room/background fragment counts, linkage, and `BRK` preview summaries for supported rooms; it does not emit the projected fragment-zone cell dump that the viewer-local comparison panel uses, and `11/10` failing there now reflects the guarded product boundary rather than a regression.
- The canonical `SCENE.HQR[2]` / `LBA_BKG.HQR[2]` pair is now a trap for fragment work if you only look at `gri_header.my_grm` or `grm_entry_index`. Background `2` reports `my_grm = 0` and `grm_entry_index = 149`, but the checked-in asset probes show that room `2` owns zero fragment entries and scene `2` has zero `grm` zones, so a fragment-focused viewer slice on `2/2` must surface an explicit zero-fragment state instead of inventing mutation overlays from the base GRM cursor alone.
- After the branch-B scene-life guard landed in `loadRoomSnapshot`, `SCENE.HQR[2]` / `LBA_BKG.HQR[2]` is no longer a valid positive runtime fixture. Keep `2/2` as a negative unsupported-scene-life case, not as the supported runtime/load baseline.
- `44/2` is only a guarded negative life-boundary case, not an unchecked interior-room topology candidate. If you skip scene-life validation on the test-only room path, scene `44` immediately falls through to `ViewerSceneMustBeInterior`, so topology evidence scans should exclude it instead of treating every guarded negative pair as another unchecked room snapshot.
- `SCENE.HQR[11]` / `LBA_BKG.HQR[10]` still matters as checked-in fragment evidence, but it is not a guarded runtime-safe load because scene `11` still crosses unsupported life outside the current branch-B boundary. Preserve that pairing only on explicit evidence or test-only unchecked paths unless the product boundary changes.
- `SCENE.HQR[11]` has more than one unsupported life blob, but the guarded scene-level boundary fails on the first one in owner order: object `12` hits `LM_DEFAULT` at byte offset `38`, while object `18` reaches a later `LM_END_SWITCH` at offset `84`. When you prove why `11/10` is guarded-negative, use `validateSceneLifeBoundary*` or assert the first-hit ordering explicitly instead of citing any unsupported blob from the scene.
- `inspect-room` now shares the guarded viewer/runtime seam. If you need the `11/10` fragment evidence pair, use the explicit test-only unchecked loader path instead of assuming the canonical CLI probe should bypass scene-life rejection.
- Fragment-bearing `grm` zones do not use the same max-coordinate convention as the older zero-fragment viewer boundary. The checked-in positive pair `SCENE.HQR[11]` plus `LBA_BKG.HQR[10]` validates only if `grm` zone maxima are treated as boundary-aligned cell endpoints, so viewer-local fragment projection must compute width/depth/height as `((max - min) / unit) + 1` instead of reusing the inclusive `max - min + 1` span rule from other debug overlays.
- The canonical `SCENE.HQR[2]` / `LBA_BKG.HQR[2]` pairing is not the whole interior background loader story. Classic interior loading also depends on `LBA_BKG.HQR[0]` as `T_BKG_HEADER` plus a late `TabAllCube` entry at `BkgHeader.Brk_Start + BkgHeader.Max_Brk`, so do not design a background decoder as if room `2` were just one self-contained HQR entry with no global indirection.
- `port/src/assets/hqr.zig` now has two HQR index spaces on purpose. The older `extractEntryToBytes` / `decodeEntryToBytes` helpers follow the repo's existing one-based view that skips classic slot `0`, while `extractClassicEntryToBytes` / `decodeClassicEntryToBytes` use the engine's zero-based `Load_HQR` numbering. Use the classic helpers for `LBA_BKG.HQR[0]` and any source-faithful loader work; mixing those spaces will shift you onto the wrong payload.
- `zig build test` is currently a real-asset regression gate, not a hermetic unit-test-only pass. The scene tests resolve the canonical extracted asset tree by default, and `port/build.zig` hard-fails if the repo-local SDL2 install is missing, so do not assume the test step can run on a stripped-down workspace.
- `zig build test` also does not prove the executable targets compile. The test step can pass while `zig build run` still fails on app-only code paths, so keep the prompt's explicit `zig build run -- ...` or `zig build tool -- ...` command in the acceptance pass whenever runtime code changed.
- Interrupted viewer launches can leave `port/zig-out/bin/lba2.exe` locked by a stale `lba2.exe` process. If the next `zig build run` fails with `AccessDenied` while updating that file, check for leftover `lba2` processes and kill them before assuming the new code failed to build.
- `tools/codex_memory.py` validates the whole v2 tree, including subsystem-pack byte budgets, before it appends history. If an `add-task-event` or other `add-*` write fails unexpectedly, check for oversized packs like `docs/codex_memory/subsystems/architecture.md` or `backgrounds.md`, not just JSONL drift.
- `scripts/dev-shell.ps1` lives at the repo root, not under `port/`. If a prompt tells you to validate from inside `port/`, either load the dev shell before `cd port` or call it as `..\\scripts\\dev-shell.ps1`; `.\scripts\dev-shell.ps1` will fail once your working directory is `port\`.
- `SCENE.HQR` giver zones do not encode one fixed bonus kind at load time. `Info0` is a bonus-flag mask that `WhichBonus()` resolves later from runtime state, so a typed decoder should expose the allowed bonus kinds or preserve the raw mask instead of inventing a singular `bonus_kind`.
- `SCENE.HQR[19]` zone bounds are not floor-truth mapping anchors. Even when their `y_min` / `y_max` stay on the 256-unit height grid, the checked-in room snapshot only proves trigger volumes, and camera-zone semantic anchor fields are not a source-backed license to invent a center, corner, or floor-projection policy for runtime mapping.
- The decoded scene object array is intentionally shorter than `object_count`. `scene/parser.zig` reads the classic `object_count` field, rejects `0`, and then allocates `object_count - 1` drawable objects because the count includes the reserved hero slot. Use `scene.objects.len` for actual object iteration or render sizing, and treat `object_count` as the source-format header field instead of a ready-to-draw array length.
- `COMMON.H` plus `GERETRAK.CPP` is not a complete track-opcode layout oracle on its own. Real scene `44` hero-track bytes decode repeated `TM_REM` opcodes even though `DoTrack` does not spell out a `TM_REM` switch case, and the checked-in runtime source still leaves `TM_NOP`, `TM_DO`, and `TM_AFF_TIMER` without explicit operand coverage there. Treat the structural decoder as the current byte-level truth and do not overclaim stronger semantics for those opcodes without new evidence.
- `COMMON.H` is not a live life-opcode oracle either. It inventories named `LM_*` ids that `GERELIFE.CPP` never handles (`LM_NOP`, `LM_ENDIF`, `LM_REM`, `LM_DEFAULT`, `LM_END_SWITCH`, `LM_SPY`, `LM_DEBUG`, `LM_DEBUG_OBJ`), keeps one dead `LM_MESSAGE_CHAPTER` case commented out in the switch, and even has `LF_*` helpers whose comparison widths are not symmetric (`LF_COL_DECORS` vs. `LF_COL_DECORS_OBJ`, `LF_FUEL`). The checked-in life evidence memo can drift too: when `life_program.zig` landed, `GERELIFE.CPP` still had four live object-scoped opcodes the memo had missed (`LM_STOP_L_TRACK_OBJ`, `LM_RESTORE_L_TRACK_OBJ`, `LM_SAVE_COMPORTEMENT_OBJ`, `LM_RESTORE_COMPORTEMENT_OBJ`). The offline `audit-life-programs` command proves the blocker is wider than scene `2` hero alone: the selected canonical scene set hits unsupported `LM_DEFAULT` in scene `2` hero plus scene `44` objects `2` and `3`, and it hits unsupported `LM_END_SWITCH` in scene `5` hero and scene `44` hero. The broader `--all-scene-entries` audit then narrows that blocker set again: across all `221` non-header scene entries and `3109` life blobs, only `LM_DEFAULT` and `LM_END_SWITCH` appear, while the other six named-but-unimplemented ids do not occur in the current asset tree. A follow-up source pass found only one nearby checked-in clue for those switch-family ids beyond the header names: the `LM_BREAK` comment `saute au END_SWITCH`. That comment is not enough to assign structural layout or runtime behavior to `LM_DEFAULT` or `LM_END_SWITCH`. Build any future life decoder from `DoLife` + `DoFuncLife` + `DoTest`, re-check the live switch for exhaustive coverage, and reject unsupported ids instead of trusting header names, comments, or older summaries alone.
- Real asset switch blocks make the `LM_BREAK` comment even weaker than it looks. In checked-in scene life bytes, `CASE` and `BREAK` jump targets line up with the first byte after `LM_END_SWITCH`, not with the `LM_END_SWITCH` opcode byte itself, so the classic `saute au END_SWITCH` note is only a rough control-flow hint. Do not use that comment to claim `LM_END_SWITCH` has an executable runtime case, to infer its byte size from the jump destination wording alone, or to assume `BREAK` lands on the marker rather than just beyond it.
- External reimplementations are not branch-A proof for unsupported life opcodes. `idajs` duplicates the same `LM_DEFAULT` / `LM_END_SWITCH` defines and inherits the same missing live `DoLife` cases, so it does not widen the primary-source evidence set. `lba2remake` goes further and implements project-local `DEFAULT` and `END_SWITCH` handlers plus parser stack bookkeeping, but that repo is still a separate reimplementation rather than a checked-in classic source oracle. Treat `lba2remake`'s zero-operand `DEFAULT` no-op and `END_SWITCH` switch-stack pop as hypothesis material only, not as canonical structural proof safe enough to promote into the Zig decoder or reopen the branch-B boundary without new checked-in primary-source evidence.
- The preserved LBArchitect room and grid specs are valuable, but they are not a byte-perfect oracle. They mix human-facing entry ranges with zero-based header indices, and the room-format note's last documented `LBA_BKG.HQR` header DWORD (`0x18`) does not match the current asset-backed regression (`86875` in the text vs. `86675` in the checked-in test for canonical background `2`). Use the docs for structure, but let current asset-backed probes and tests win for exact numeric values.
- `docs/MEMSYS.md` or older notes can lag behind the current memory implementation. Treat `docs/codex_memory/README.md`, `project_brief.md`, `current_focus.md`, the subsystem packs, and the v2 CLI as the canonical memory contract.
- Historical migration notes are not an authority over final pack placement. When an older migration memo disagrees with the checked-in v2 packs and typed history, trust the checked-in v2 tree.
- The checked-in v2 history files are not guaranteed to pass `python tools/codex_memory.py validate`. If the memory CLI starts failing before you change anything, inspect `docs/codex_memory/task_events.jsonl` for stale `record_id` hashes, fractional-second timestamps, or overlong summaries instead of assuming the tool is broken.
- The LM task tracker uses `LM_TASKS/LM_CURRENT_TASK.md`, not `LM_TASKS/CURRENT_TASK.md`. If an LM prompt says `CURRENT_TASK.md`, verify the actual file name before assuming the tracker is missing.
