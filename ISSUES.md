# Common Agent Traps

- A stale ignored `port/hello-world/` directory can survive on disk after the real Phase 1 workspace is moved to `port/`. If Phase 1 docs claim the placeholder was replaced but the directory still exists locally, delete it instead of preserving the obsolete layout.
- `sidequest/` and `LM_TASKS/` are independent workstreams, not part of canonical memory pickup. Do not treat them as execution owners unless a prompt explicitly widens scope.
- `python3 tools/codex_memory.py context --include-history ...` still hides excluded durable history by default. If a task explicitly needs `sidequest/`, `LM_TASKS/`, LM/debug streams, or other noncanonical history, add `--include-excluded-history` instead of assuming the records disappeared.
- The checked-in memory files can lag behind the live worktree. Verify the code and `git status` before treating `docs/codex_memory/current_focus.md` or any subsystem pack as a statement about dirtiness, generated outputs, or active experiments.
- Some Codex worktrees can have a stale `.git` pointer even when the filesystem checkout is usable. If `git status` fails with `not a git repository` under `.codex/worktrees/...`, treat the tree as a plain filesystem checkout until the worktree metadata is repaired instead of assuming the repo content is missing.
- A `.codex/worktrees/...` checkout may not mirror bulky original-runtime artifacts under `work/`. If a helper needs extracted `LBA2.EXE` assets, verify the canonical path under `D:\repos\reverse\littlebigreversing\work\...` instead of assuming the current worktree carries the same files.
- `docs/PORTING_REPORT.md`, `docs/MEMSYS.md`, and older migration notes are historical context, not the current implementation contract. Use `docs/LBA2_ZIG_PORT_PLAN.md`, `docs/codex_memory/`, and the live `port/` tree for current state.
- The canonical current-state owners are `docs/LBA2_ZIG_PORT_PLAN.md`, `docs/codex_memory/current_focus.md`, and the relevant subsystem packs. Do not recreate a separate prompt-brief owner for the next slice.
- `tools/codex_memory.py context --path ...` only works for repo paths covered by `docs/codex_memory/subsystems/INDEX.md`. Legitimate current-state code paths such as `port/src/app/viewer_shell.zig` still fail with `no subsystem mapping for path` until the index is expanded, so use `--subsystem` or a mapped owner path instead of assuming path-scoped retrieval is repo-wide.
- Raw `SCENE.HQR` entry bytes are not the runtime scene payload. The classic loader reads an HQR `COMPRESSED_HEADER` first and may LZ-decompress the entry before parsing scene fields, so typed decoders must decode the resource block before interpreting world/object/zone data.
- Scene numbering is easy to misread. The classic loader calls `Load_HQR(..., numscene + 1)` because `SCENE.HQR[0]` is reserved for `SizeCube.MAX`, while the current `inspect-scene` command accepts the raw one-based HQR entry index. Keep semantic scene numbers and HQR entry numbers separate.
- The raw-to-loader mapping has two offsets in this repo, not one. Classic code uses raw HQR slot `0` for `SizeCube.MAX`, while the Zig tooling exposes physical entries starting at `1`, so `SCENE.HQR[2]` is classic loader scene `0` and `classic_loader_scene_number` must use `entry_index - 2` for real scenes.
- `inspect-room-intelligence` no longer reads the legacy LBArchitect `.hqd` files at runtime. Room names now come from the checked-in generated Zig module at `port/src/generated/room_metadata.zig`, so update labels by rerunning `py -3 .\tools\generate_room_metadata.py` instead of patching the runtime resolver or reviving the old text-file parser.
- The MBN `SCENE.HQR` lists use type markers that matter. In post `177798`, `~1` means isometric room data and `~2` means outside scene data; keep the type marker with the descriptor instead of copying only the human-readable scene name.
- The checked-in MBN corpus index is broader than the raw evidence layer. A topic can appear in `docs/mbn_reference/corpus/index/topics_discovered.jsonl` and `topic_classification.jsonl` without a matching raw post payload or any `tools/mbn_workbench.py search-evidence` hits.
- The shell-detection snippet in `AGENTS.md` is misleading under Bash because it uses PowerShell redirection syntax on the Bash side. Treat the current shell as Bash from the provided environment context or use a Bash-safe probe instead.
- Parked nested Git metadata under `reference/discourse-downloader/.git.disabled/` and `reference/littlebigreversing/.git.disabled/` is currently tracked by the parent repo. Treat those `objects/`, `refs/`, `logs/`, `HEAD`, and `config` files as portability noise, not as canonical project state that needs to be synchronized across machines.
- On this Windows checkout, `bash -lc 'git status'` under `/mnt/d/...` can report a false-positive mass-dirty worktree while native Windows `git status` shows only the real local edits. For repo-state or portability checks, trust native Git from the provided PowerShell shell before assuming hundreds of files changed.
- In the PowerShell-hosted workflow, `bash -lc "..."` can still be mangled before Bash sees it when the payload contains nested double quotes or PowerShell-sensitive tokens. Prefer single-quoted Bash payloads in tool calls so the inner command reaches Bash unchanged.
- On the current Windows setup, reusing the checked-in Ghidra project from a `.codex/worktrees/...` path is not a reliable `ghb` spike target. Ghidra can report `project locked` or `IllegalArgumentException` path-element errors there, so prefer a disposable project outside the Codex worktree for fresh static-analysis automation.
