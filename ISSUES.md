# Common Agent Traps

- A stale ignored `port/hello-world/` directory can survive on disk after the real Phase 1 workspace is moved to `port/`. If Phase 1 docs claim the placeholder was replaced but the directory still exists locally, delete it instead of preserving the obsolete layout.
- Raw `SCENE.HQR` entry bytes are not the same thing as the runtime scene payload. The classic loader reads an HQR `COMPRESSED_HEADER` first and may LZ-decompress the entry before parsing scene fields, so typed decoders must decode the resource block before interpreting world/object/zone data.
- Scene numbering is easy to misread. The classic loader calls `Load_HQR(..., numscene + 1)` because `SCENE.HQR[0]` is reserved for `SizeCube.MAX`, while the current `inspect-scene` command accepts the raw one-based HQR entry index. Keep semantic scene numbers and HQR entry numbers separate, and do not revive the old `SCENE.HQR[4]` exterior misclassification just because that entry decodes cleanly.
- The raw-to-loader mapping has two offsets in this repo, not one. Classic code uses raw HQR slot `0` for `SizeCube.MAX`, while the Zig tooling exposes physical entries starting at `1`, so `SCENE.HQR[2]` is classic loader scene `0` and `classic_loader_scene_number` must use `entry_index - 2` for real scenes.
- The MBN `SCENE.HQR` lists use type markers that matter. In post `177798`, `~1` means isometric room data and `~2` means outside scene data; if you pick a golden exterior target from that list, keep the type marker with the descriptor instead of copying only the human-readable scene name.
- The checked-in memory files can lag behind the actual worktree state. As of 2026-03-19, `docs/codex_memory/current_focus.md` and `docs/codex_memory/handoff.md` still describe the repo as intentionally dirty in corpus/reference areas, but `git status --short` is clean. Verify the real worktree before treating the dirty-worktree warning as current fact.
- `docs/codex_memory/handoff.md` can reference architecture docs that are not actually present in `docs/architecture/`. As of 2026-03-26, it points at `docs/architecture/current_state_and_minimal_playable_port.md`, but that file is missing from the worktree. Verify the file exists before treating it as canonical context, and recreate or fix the reference instead of assuming the architecture doc set is complete.
- The shell-detection snippet in `AGENTS.md` is misleading under Bash. `pwsh -c 'Write-Host "PowerShell"' 2>$null || echo "Linux Bash"` uses PowerShell redirection syntax on the Bash side, so running it literally from Bash emits `/bin/bash: $null: ambiguous redirect` before falling through. Treat the current shell as Bash from the provided environment context or use a Bash-safe probe instead of trusting that exact command output.
- `docs/PORTING_REPORT.md` still carries a historical `Current Workspace Status` section that talks about an earlier CMake-first setup path and environment steps. Treat that report as feasibility and evidence context only; use `docs/LBA2_ZIG_PORT_PLAN.md`, `docs/codex_memory/`, and the live `port/` tree for the actual current implementation state.
- `docs/PROMPT.md` can lag behind the durable memory and still point at a slice that `docs/codex_memory/task_log.jsonl` already records as completed. Before executing the prompt literally, cross-check it against `handoff.md`, `decision_log.jsonl`, and the latest task events so you do not repeat a completed evidence pass as if it were still the next step.
- The canonical `SCENE.HQR[2]` / `LBA_BKG.HQR[2]` pairing is not the whole interior background loader story. Classic interior loading also depends on `LBA_BKG.HQR[0]` as `T_BKG_HEADER` plus a late `TabAllCube` entry at `BkgHeader.Brk_Start + BkgHeader.Max_Brk`, so do not design a background decoder as if room `2` were just one self-contained HQR entry with no global indirection.
- `port/src/assets/hqr.zig` now has two HQR index spaces on purpose. The older `extractEntryToBytes` / `decodeEntryToBytes` helpers follow the repo's existing one-based view that skips classic slot `0`, while `extractClassicEntryToBytes` / `decodeClassicEntryToBytes` use the engine's zero-based `Load_HQR` numbering. Use the classic helpers for `LBA_BKG.HQR[0]` and any source-faithful loader work; mixing those spaces will shift you onto the wrong payload.
- `zig build test` is currently a real-asset regression gate, not a hermetic unit-test-only pass. The scene tests resolve the canonical extracted asset tree by default, and `port/build.zig` hard-fails if the repo-local SDL2 install is missing, so do not assume the test step can run on a stripped-down workspace.
- `SCENE.HQR` giver zones do not encode one fixed bonus kind at load time. `Info0` is a bonus-flag mask that `WhichBonus()` resolves later from runtime state, so a typed decoder should expose the allowed bonus kinds or preserve the raw mask instead of inventing a singular `bonus_kind`.
- `COMMON.H` plus `GERETRAK.CPP` is not a complete track-opcode layout oracle on its own. Real scene `44` hero-track bytes decode repeated `TM_REM` opcodes even though `DoTrack` does not spell out a `TM_REM` switch case, and the checked-in runtime source still leaves `TM_NOP`, `TM_DO`, and `TM_AFF_TIMER` without explicit operand coverage there. Treat the structural decoder as the current byte-level truth and do not overclaim stronger semantics for those opcodes without new evidence.
- `COMMON.H` is not a live life-opcode oracle either. It inventories named `LM_*` ids that `GERELIFE.CPP` never handles (`LM_NOP`, `LM_ENDIF`, `LM_REM`, `LM_DEFAULT`, `LM_END_SWITCH`, `LM_SPY`, `LM_DEBUG`, `LM_DEBUG_OBJ`), keeps one dead `LM_MESSAGE_CHAPTER` case commented out in the switch, and even has `LF_*` helpers whose comparison widths are not symmetric (`LF_COL_DECORS` vs. `LF_COL_DECORS_OBJ`, `LF_FUEL`). The checked-in life evidence memo can drift too: when `life_program.zig` landed, `GERELIFE.CPP` still had four live object-scoped opcodes the memo had missed (`LM_STOP_L_TRACK_OBJ`, `LM_RESTORE_L_TRACK_OBJ`, `LM_SAVE_COMPORTEMENT_OBJ`, `LM_RESTORE_COMPORTEMENT_OBJ`). The offline `audit-life-programs` command proves the blocker is wider than scene `2` hero alone: the selected canonical scene set hits unsupported `LM_DEFAULT` in scene `2` hero plus scene `44` objects `2` and `3`, and it hits unsupported `LM_END_SWITCH` in scene `5` hero and scene `44` hero. The broader `--all-scene-entries` audit then narrows that blocker set again: across all `221` non-header scene entries and `3109` life blobs, only `LM_DEFAULT` and `LM_END_SWITCH` appear, while the other six named-but-unimplemented ids do not occur in the current asset tree. A follow-up source pass found only one nearby checked-in clue for those switch-family ids beyond the header names: the `LM_BREAK` comment `saute au END_SWITCH`. That comment is not enough to assign structural layout or runtime behavior to `LM_DEFAULT` or `LM_END_SWITCH`. Build any future life decoder from `DoLife` + `DoFuncLife` + `DoTest`, re-check the live switch for exhaustive coverage, and reject unsupported ids instead of trusting header names, comments, or older summaries alone.
